## Part I HTTP의 이해
<strong>HTTP란?</strong><br>
아주 쉽게 이해 말 하자면 `HTTP(HyperText Transfer Protocol)`는 Web Browsers와 Servers가 서로 커뮤니케이션이 가능한 언어 입니다. 그럼 전세계의 컴퓨터들은 어떻게 서로 커뮤니케이션을 하는지 알아 보겠습니다. 요즘 우리가 사용 하는 웹브라우저는 Internet Explorer, Google Chrome, Mozilla Firefox, Safari, Opera, 등 이 있습니다. 웹브라우저를 열고 방문 하고 싶은 웹 주소를 입력창에 입력 하게 될 겁니다. 예를 들어 https://www.google.com/ 주소를 입력 합니다. 이때 본인 컴퓨터는 다른 커뮤니케이션이 시작 됩니다.<br>

우리가 웹브라우저에서 https://www.google.com/ 주소를 입력 후 Enter key를 누른면 어떻게 반응이 되는지 알아 보겠습니다. Enter Key가 누르게 되면 컴퓨터는 다른 컴뷰터와 커뮤니케이션을 시작 하게 됩니다. 커뮤니케이션을 하는 다른 컴뷰터를 Server라고 부르고 이 Server는 아마도 수천 마일 떨어져 있는 위치에 있을 것 입니다. 백만 초의 시간으로 우리의 웹브라우저는 Server 한테 원하는 웹사이트를 요청 하게 되는데 여기서 클라이언트가 주소창을 통해 서버로부터 Server는 HTTP라는 언어로 요청을 하게 되고 요청의 대한 응답으로 서버는 클라이언튼에게 HTTP 응답을 하게 됩니다.<br>

웹 서버의 리소스는 각자 이름을 가지고 있는데 이 리소스를 찾는 역할을 하는 것이 바로URL(Uniform Resource Locator)입니다. 즉, URL로 리소스를 찾고 다시 반환 하는 것 입니다.<br>

<strong>HTTP Statelsss와 Stateful차이</strong><br>
`Stateless`란 http와 같이 client의 이전 상태를 기록하지 않는 접속이란 의미입니다.
그에 비해 `Stateful`은 client의 이전 상태를 기록하고 있는 것입니다. 즉, `Stateless`는 웹서버가 사용자의 작업을 기억하고 있지 않다는 의미이고 `Stateful`은 사용자의 상태를 서버가 기억하고 있다가 유용한 정보로써 활용한다는 것입니다.<br>

<strong>HTTP Methods</strong><br>
1. <ins>Get</ins><br>
메소드는 특정 리소스의 표시를 요청합니다. GET을 사용하는 요청은 오직 데이터를 받기만 합니다.

2. <ins>Head</ins><br>
HEAD 메소드는 GET 메소드의 요청과 동일한 응답을 요구하지만, 응답 본문을 포함하지 않습니다.

3. <ins>Post</ins><br>
POST 메소드는 특정 리소스에 엔티티를 제출할 때 쓰입니다. 이는 종종 서버의 상태의 변화나 부작용을 일으킵니다.

4. <ins>Put</ins><br>
PUT 메소드는 목적 리소스 모든 현재 표시를 요청 payload로 바꿉니다.

5. <ins>Delete</ins><br>
DELETE 메소드는 특정 리소스를 삭제합니다.

6. <ins>Connect</ins><br>
CONNECT 메소드는 목적 리소스로 식별되는 서버로의 터널을 맺습니다.

7. <ins>Options</ins><br>
OPTIONS 메소드는 목적 리소스의 통신을 설정하는 데 쓰입니다.

8. <ins>Trace</ins><br>
TRACE 메소드는 목적 리소스의 경로를 따라 메시지 loop-back 테스트를 합니다.

9. <ins>Patch</ins><br>
PATCH 메소드는 리소스의 부분만을 수정하는 데 쓰입니다.

## Part II HTML의 이해

<strong>HTML(HyperText Markup Language)</strong><br>
HTML은 웹 브라우져에게 웹 페이지가 어떤 식으로 표현될지를 알려주는 언어라고 생각하시면 됩니다. 위키피디아처럼 단순하고 아주 큰 문서를 떠올리면 HTML 이라는 언어를 이용해서 타이틀을 크고 진하게 만들 수 있고 알맞은 서체를 정할 수 있고 다른 웹페이지로 연결되는 기능을 줄 수 있으며 어떤 글자들을 굵게 나타내거나 이텔릭체로 표현할 수도 있고 페이지 중간에 그림을 넣고 그림들을 오른쪽이나 왼쪽에 줄맞춤 시킬 수도 있습니다.<br>

하지만 그림이나 동영상 같은 다른 것들은 자신의 URL을 가지고 있는 분리된 자료들입니다. 요청을 해야 얻을 수 있는 것들 입니다. 브라우져는 이런 자료들 각각에 HTTP 요청을 보내서 그들이 도착하면 화면에 보여줍니다. <br>

웹 페이지에 아주 많은 다른 그림들이 이용된다면 그들 각각의 HTTP요청을 처리하느라 페이지를 띄우는 시간이 더 느려지게 됩니다. 여러분이 웹을 이용하다보면 단순히 GET 요청만 하지는 않습니다. 가끔은 정보를 보내기도 합니다. 서식을 작성하거나 검색을 할 때 처럼 말입니다. 여러분의 브라우져들은 이런 정보들을 단순한 글자의 형태로 HTTP 호스트 요청을 이용해서 웹서버에 보냅니다. 예를 들어 google에 로그인을 한다고 해봅니다. 제일 먼저 일어나는 일은 POST 요청을 하는 것입니다. Google의 로그인 페이지에 정보들을 포함시켜서 발송하는 것입니다. 그 정보에는 당신의 이메일 주소 비밀번호가 있습니다. 이는 Google의 서버로 보내집니다. 그러면 서버는 이를 검토해서 이사람은 좋아 누구누구 이군 이라고 알아낼 겁니다. 그러면 서버는 당신의 웹 브라우져에게 페이지를 보내서 성공입니다 누구누구으로 로그인 되었습니다 하고 알려주게 됩니다.<br>

하지만 이런 웹페이지 외에도 보이지 않은 쿠키 데이터가 함께 따라가서 여러분의 브라우져를 통해 보고 저장할 수 있도록 할 겁니다. 쿠키는 웹사이트가 여러분이 누구인지를 기억하는 유일한 수단이기 때문에 매우 중요합니다. 모든 쿠키 정보는 Google에 대한 ID카드와 같습니다. 그것은 당신이 누구임을 확인시켜부는 번호 입니다. 당신의 웹 브라우져가 그 번호를 갖고 있으면 Google를 빠져나갔다가 다음 번에 Google사이트에 재접속할 때 당신의 웹 브라우져는 자동적으로 ID 넘버를 보게 되고 누구으로부터 온 요청인 것을 알게 될 것입니다. <br>

인터넷은 완전히 개방적입니다. 모든 연결이 공유되고 있고 평범한 글자로 정보가 전달됩니다. 그래서 해커들이 인터넷을 통해 전달되는 개인 정보들을 빼내갈 수 있습니다. 하지만 안전한 웹사이트들은 웹 브라우져가 Secure Sockets Layer이나 Transport Layer Security 같은 것을 이용하는 보안 채널을 통해 소통하도록 함으로써 해킹을 방지하고 있습니다. <br>

SSL과 TLS는 여러분의 통신을 둘러싼 보안층으로 정보가 새나가지 않도록 보호해 준다고 생각하시면 됩니다. SSL과 TLS 가 작동하는 지는 여러분 브라우져의 웹주소 입력란의 HTTPS 옆에 작은 자물쇠가 있는지를 보시면 알 수 있습니다 .HTTPS 통신방식은 여러부느이 HTTPS 요청이 안전하고 보호 되고 있다는 것을 보증하고 있습니다. 웹 사이트가 여러분의 브라우져에게 보안된 채널로 연결시키려고 하면 웹사이트는 전자 인증이라는 것을 제공하는데 이 웹사이트는 허위 사이트가 아니라는 것을 보증하는 ID 카드 같은 것입니다. 
전자 서명은 인증 기관에 의해 발급된다. 인증 기관은 정부가 ID 나 여권을 발급할 수 있는 것처럼 웹사이트들의 진위성을 확인하고 그들에 대한 인증서를 발급하는 역할을 합니다. 제대로 발급된 전자 인증서 없이 웹사이트가 보안 연결을 시작하려 하면 여러분의 브라우져가 경고를 줄 것입니다. 이것은 웹 브라우징의 기초로 우리가 매일매일 보고있는 인터넷의 일부 입니다. <br>

요약하자면 HTTP와 DNS는 HTML, 미디어 파일과 같이 웹상에 존재하는 것들의 송수신을 관리합니다. 그리고 이것을 가능하게 한 핵심적인 기술은 TCP/IP와 라우터 네트워크라는 작은 단위로 정보를 쪼개서 전달하는 것입니다. 이런 단위들은 0과1의 이진수 배열로 이루어져 있고 전선이나 광섬유 무선 네트웤을 통해 전달됩니다. 다행스럽게도 여러분이 일단 한 층의 인터넷이 어떻게 작동하는지 알게되면 전체과정을 속속들이 기억하지 않아도 신뢰할 수 있게 됩니다. 이런 시스템을 이루는 여러 층들이 함께 일하여 믿을 수 있는 환경에서 안전하게 정보를 전달하는 것입니다. <br>

<strong>Secure Sockets Layer(SSL)와 Transport Layer Security(TLS)란?</strong><br>
Secure Sockets Layer(SSL)는 Netscape(네스케이프)에 의해 발명 되었고 웹서버와 브라우저 사이의 보안을 위해 만들어 졌습니다.<br>
Transport Layer Security(TLS)는 SSL를 점차 폭넓게 사용되다가 표준화 기구인 IETF(Internet Engineering Task Force)의 관리로 변경되면서 TLS라는 이름으로 변경 됬습니다. TLS 1.0은 SSL 3.0을 계승합니다. 하지만 TLS라는 이름보다 SSL이라는 이름이 훨씬 많이 사용되고 있습니다.<br>
 
<strong>Internet Protocol(IP)란?</strong><br>
 전 세계 컴퓨터가 인터넷을 하기 위해서는 서로의 정체를 알 수 있도록 특별한 주소를 부여했는데 이 주소를 IP주소라고 합니다. IP는 4개의 숫자로 구성되며 숫자의 크기에 따라 IPv4(32비트, 각 숫자는 1바이트), IPv6(128비트, 각 숫자는 4바이트)로 나눠 집니다. 일반적으로 IPv4는 10진수로 표현하고 각 자리는 .으로 구분하고 IPv6는 각자리를 4자리 14진수로 표현하고 각 자리는 :로 구분합니다. <br>

<img src="/assets/images/ip.png"> <br>

<strong>Transmission Control Protocol(TCP)</strong><br>
IP로 컴퓨터의 위치를 찾았다면 원래의 목적인 정보를 전달해야 합니다. 인터넷은 패킷을 이용해서 정보를 전달하므로 이 패킷을 전달하는 규약도 필요 합니다. 패킷은 데이터를 일정한 크기로 자른 단위로 인터넷에서 정보를 전달하는 단위입니다. 나누어진 패킷이 순서대로 도착한다는 보장이 없으므로 규칙이 필요합니다. TCP는 전달받은 패킷을 재조립하고, 패킷에 손상이 있거나 손실된 패킷이 있다면 재전송을 요청하는 패킷을 전송하여 재전송 받습니다. TCP는 데이터의 흐름을 저어할 수는 있지만 데어터가 전송되어야 할 경로에 대한 규약은 아니기 때문에 경로에 대한 규약인 IP를 함께 사용해야 합니다. IP와 TCP는 반드시 함께 사용되어야 합니다.

<strong>HTML 실제참조(character entity references) 와 문자참조(numerical character references) 차이</strong><br>
HTML/XHTML 상의 특수기호가 그대로 표시되도록 기술하거나 키보드로 직접 입력할 수 없는 문자를 기술하기 위한 방법을 문자 참조(character referendes) 라고 합니다.

## Part III Open ID and Oauth의 이해

<strong> OpenId란?</strong><br>
하나의 ID로, 한번의 로그인으로 여러 서비스를 일일이 가입할 필요없이 사용하게 하는 인증 서비스 표준입니다. <br>

이게 가능한 이유는 `OpenID`에서 ID 는 인터넷 주소 (URL) 로 표현됩니다. 인증을 제공 해주는 서비스를 통해서 사용자가 ID 를가지고 있음을 증명하게 됩니다. URL 이기 때문에 가지는 장점중 하나는 인터넷 전체에서 고유한 주소이기 때문에, 지원하는 어떤 사이트에서도 반복적인 가입절차 없이도 나를 구분할 수 있게 해줍니다. 이러한 식별을 통해서 쉽게 개인화된 서비스를 제공 받을 수 있습니다.<br>

<img src="/assets/images/openID.JPEG"><br>

<strong> Oauth란?</strong><br>
`Oauth`는 `Open Authorization` 또는 `Open Authentication`의 뜻하는 것으로 애플리케이션인 페이스북,구글,트위터 Service Provider의 유저의 비밀번호를 Third party앱에 제공 없이 인증 해주고, 인가를 할 수 있는 오픈 스탠다드 프로토콜입니다.
아래와 같은 로그인 창을 보셨을 것입니다. 별도의 회원가입 없이 로그인을 제공하는 플랫폼의 아이디만 있으면 서비스를 이용 할 수 있습니다. 외부 서비스에서도 인증을 가능하게 하고 그 서비스의 API를 이용하게 해주는 것, 이것을 바로 OAuth라고 합니다. <br>

<img src="/assets/images/Oauth.JPEG"><br>

<strong>OpenID v.s. Oauth 차이</strong><br>
`OAuth 2.0`은 인증 프로토콜이 아니라 인증 프레임워크라는 점을 먼저 이해해야 합니다. OAuth 2.0은 많은 멋진 작업에 사용될 수 있는데, 그 중 하나는 사람 인증입니다.
OpenID Connect는 속성 release 및 인증을 위해 특별히 설계된 OAuth 2.0의 "Profile"입니다.
기술적 관점에서 볼 때, OpenID와 Oauth 의 큰 차이점은 id_token에 있습니다. 연합 인증에 한정되어 있기 때문에 OAuth 2.0에 정의된 id_token은 없습니다.
